## 1. 這個 Bot 能做什麼

這是一個整合了「技術知識庫」的 AI 助手，運行在你的私人 LINE 頻道上。它的特點是：

- **根據問題類型自動切換專家角色**，不需要你指定模式
- **從你的私人知識庫搜尋相關內容**，給出貼合你情境的回答
- **偵測你的情緒狀態**，調整回覆語氣與長度
- **回覆格式因問題類型而異**：操作步驟、決策比較、除錯思路各有不同格式

---

## 2. 六種專家模式（Skill）

Bot 內建六個專家角色，每次對話都會自動選擇最合適的一個。

### 技術架構師

**適合問：**
- 系統設計、API 架構、資料庫選型
- Supabase、FastAPI、LINE Bot、RAG、pgvector 相關問題
- 部署策略、工程落地方案

**回覆風格：** 可落地、有風險評估、給出具體路徑（檔案路徑、schema、API contract）

**範例問法：**
> 「pgvector 和 pg_trgm 的 hybrid search 怎麼設計？」
> 「FastAPI 的 lifespan 怎麼管理 Supabase client？」

---

### 資料科學家

**適合問：**
- 分析方法、特徵工程、A/B 實驗設計
- 模型評估、指標設計與解讀
- 統計推論與因果分析

**回覆風格：** 先定義問題與評估指標，區分描述/推論/預測，給可驗證步驟

**範例問法：**
> 「這個分類模型 AUC 0.72，算好嗎？」
> 「怎麼設計一個 proper holdout set？」

---

### 商業策略師

**適合問：**
- 商業模式設計、產品定位
- 定價策略、go-to-market
- 市場切入分析與策略比較

**回覆風格：** 定義目標客群與限制條件，呈現取捨與風險，提供決策框架

**範例問法：**
> 「SaaS 產品初期應該免費還是收費？」
> 「如何評估一個市場值不值得進入？」

---

### 哲學辯證者

**適合問：**
- 概念澄清與定義比較
- 價值觀衝突、倫理問題
- 邏輯推演與多角度辯證

**回覆風格：** 先定義核心概念，區分事實/價值/策略判斷，提出反例，最後給一個可執行建議

**範例問法：**
> 「效率和公平可以兼顧嗎？」
> 「做產品和做藝術在本質上有什麼不同？」

---

### 情緒校準者

**適合問：**
- 感到焦慮、迷失方向、不確定自己的選擇
- 懷疑作品或努力的價值
- 需要冷靜整理思路，而不是被安慰

**回覆風格：** 不灌雞湯，先承認現實壓力，把情緒拆解成可處理的問題，給出最小下一步

**範例問法：**
> 「我最近很迷茫，不知道要往哪個方向走」
> 「我做了很多事情，但感覺都沒有進展」

---

### 一般對話

**適合問：**
- 打招呼、閒聊
- 簡單的常識問題
- 沒有明顯專業領域的問題

**回覆風格：** 簡潔、友善，不裝作知道私人知識

**範例問法：**
> 「你好！」
> 「今天天氣怎樣？」

## 3. Bot 怎麼理解你的問題

每條訊息進來，Bot 會依序完成以下判斷：

```
你的訊息
  ↓
意圖分析：選擇哪個 Skill？需不需要查知識庫？
  ↓
（如需要）知識庫搜尋：向量搜尋 + 關鍵字搜尋，合併排序
  ↓
生成回覆：依 Skill 角色、回覆格式、情緒狀態組合 prompt
  ↓
送出 LINE 訊息
```

整個過程通常在 **3～8 秒**內完成。

## 4. 回覆格式說明

Bot 會根據問題類型自動選擇最合適的回覆格式，不需要你指定。

| 格式 | 什麼時候出現 | 特徵 |
|------|------------|------|
| **簡答** | 打招呼、簡單問題 | 3 句以內，無條列 |
| **結構化** | 技術說明、多步驟概念 | 有條列（1. / -），複雜時有小節標題 |
| **逐步操作** | 「怎麼做」「幫我一步一步」 | 嚴格序號步驟，最後有完成確認 |
| **決策輔助** | 「A 還是 B」「該選哪個」 | 選項比較 → 明確建議 → 風險提示 |
| **除錯思路** | 「錯誤」「壞掉了」「找不到原因」 | 可能原因 → 驗證方式 → 修法 |
| **反思引導** | 情緒性問題、價值觀探討 | 無條列，有問句，短，結尾才給建議 |

## 5. 情緒感知功能

Bot 會從你的文字偵測情緒狀態，並調整回覆策略：

| 偵測到的狀態 | Bot 的應對方式 |
|------------|--------------|
| **焦慮** | ≤ 3 句，只給 1 個具體行動，加鼓勵句 |
| **挫折** | 先承認問題確實麻煩，給最小可行的下一步 |
| **緊迫** | 直接給最重要的一個步驟，省略背景說明 |
| **困惑** | 從最基本概念開始，一次只解釋一件事 |
| **好奇** | 語氣輕鬆，可延伸探索方向 |
| **沉思** | 先以問句引導，不急著給答案 |
| **中立** | 依問題類型的格式正常回覆 |

> **提示**：如果你「很急」或「很焦慮」，把這個狀態直接說出來，Bot 的回覆會更精準、更簡短。

## 6. 知識庫與 RAG 說明

### 什麼是 RAG？

RAG（Retrieval-Augmented Generation）是指 Bot 在生成回覆前，會先從你的私人知識庫搜尋相關段落，再把搜尋結果作為上下文提供給 AI。這讓 Bot 能回答知識庫裡的私人內容，而不只是通用知識。

### 目前的知識庫內容

知識庫中包含以下類別的內容（透過 `ingest_markdown.py` 匯入）：

| 類別 | 說明 |
|------|------|
| `engineering` | 工程架構、系統設計筆記 |
| `rag` | RAG 系統學習筆記（LangGraph 等） |
| `analytics` | 資料分析、實驗設計筆記 |
| `strategy` | 商業策略、市場分析筆記 |
| `philosophy` | 哲學思考、概念辯證筆記 |
| `personal_notes` | 個人反思、工作日誌 |

### 「目前知識庫沒有足夠資料」是什麼意思？

Bot 在知識庫搜尋後找不到相關段落時會出現這個訊息。這時 Bot 仍然會盡力用通用知識回答，但回覆不會包含你的私人筆記內容。

**解法**：把相關筆記或文件匯入知識庫（詳見 [setup.md](setup.md) 的知識庫匯入章節）。

## 7. 對話範例

### 技術問題（結構化 / 逐步操作）

**你說：**
> 幫我一步一步設定 ngrok 連 LINE Webhook

**Bot 回覆（逐步操作格式）：**
> 1. 安裝 ngrok：`brew install ngrok/ngrok/ngrok`
> 2. 設定 authtoken：`ngrok config add-authtoken <你的 token>`
> 3. 啟動隧道：`ngrok http 8000`
> 4. 複製 Forwarding URL（例：`https://xxxx.ngrok-free.app`）
> 5. 在 LINE Developers Console → Messaging API → Webhook URL 填入 `https://xxxx.ngrok-free.app/webhook`
> 6. 點「Verify」確認
>
> 完成後確認：收到「Success」且 ngrok Web Inspector（`localhost:4040`）顯示 LINE 的驗證請求。

---

### 決策問題（決策輔助格式）

**你說：**
> RAG 和 fine-tuning，我該選哪個？

**Bot 回覆（決策輔助格式）：**
> **選項 A：RAG**
> 知識可隨時更新，成本低，適合私人筆記、文件問答場景。缺點：搜尋品質影響回覆品質。
>
> **選項 B：Fine-tuning**
> 適合改變模型風格或口吻，訓練一次後推論快。缺點：知識更新需重新訓練，成本高。
>
> **建議選 RAG**，因為你的需求是私人知識整合，知識內容會持續增加，RAG 的更新成本幾乎為零。
>
> **風險提示**：RAG 對知識庫分類（category）設定敏感，分類錯誤會導致搜尋命中失敗。

---

### 焦慮情緒（情緒校準 + 短回覆）

**你說：**
> 我最近很焦慮，覺得學了很多東西但什麼都沒做出來

**Bot 回覆（≤ 3 句，1 個行動）：**
> 「學了很多但沒有產出」這種感覺很常見，通常不是能力問題，而是缺少一個具體的輸出目標。
>
> 現在最小的一步：**選一個你上週學到的東西，今天用 30 分鐘寫成一段筆記或一個小 demo**。
>
> 不用完整，只要有一個可以指著說「這是我做的」的東西就夠了。

---

### 除錯問題（除錯思路格式）

**你說：**
> LINE Bot 沒有回覆，但 webhook 有收到請求，日誌也沒有錯誤

**Bot 回覆（除錯思路格式）：**
> **可能原因：**
> 1. LINE 的 Reply Token 已過期（5 秒限制），Bot 用了非同步處理但在 5 秒後才呼叫 Reply API
> 2. `reply_message()` 呼叫後有 exception 被吞掉（logger 沒用 `exception()`）
> 3. Push Message 月配額用盡（免費方案 200 則/月）
>
> **驗證方式：**
> - 確認 webhook handler 是否在 5 秒內 return 200，非同步任務是否正確觸發
> - 把所有 `except` 改為 `logger.exception()`，重現問題
> - LINE Developers Console → Messaging API → 查看 Push API 用量
>
> **修法：** 在 webhook handler 立刻 return 200，把回覆邏輯放進 background task（`asyncio.create_task`）。

## 8. 使用技巧

### 讓 Bot 選對 Skill

Bot 自動路由通常準確，但你也可以明確標示情境讓它更精準：

| 說法 | 效果 |
|------|------|
| 「從架構角度...」「系統設計上...」 | 更容易觸發技術架構師 |
| 「用資料角度分析...」 | 更容易觸發資料科學家 |
| 「我在想這件事的意義是...」 | 更容易觸發哲學辯證者 |
| 「我最近很焦慮/迷茫...」 | 更容易觸發情緒校準者 |

### 取得更精準的知識庫回覆

- 問題中包含知識庫筆記的關鍵詞時，搜尋命中率更高
- 如果回覆說「知識庫資料不足」，代表相關筆記可能還未匯入，或分類不符

### 處理超長回覆

LINE 單則訊息上限為 5000 字元，Bot 會自動分割成多則傳送。如果你想要更短的回覆，可以說：

> 「簡短說明就好」「給我最重要的一點」「3 句以內」

## 9. 已知限制

| 限制 | 說明 |
|------|------|
| **無記憶** | 每次對話都是獨立的，Bot 不記得你上次說過的內容（近期對話歷史除外） |
| **知識庫靜態** | 知識庫內容需手動匯入，Bot 不會自動抓網路資料 |
| **LINE 免費方案** | Push Message 每月 200 則上限；Reply Message 無限制 |
| **ngrok 本機測試** | 每次重啟 ngrok，URL 會改變，需更新 LINE Webhook URL |
| **回覆速度** | 查詢知識庫 + AI 生成約需 3～8 秒，網路波動時可能更長 |
| **不支援圖片/貼圖** | 目前只處理文字訊息，圖片與貼圖會被忽略 |
